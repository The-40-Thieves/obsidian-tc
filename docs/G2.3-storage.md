# Storage and Schema (G2.3)

This document specifies the obsidian-tc V1.0 storage layer: SQLite schema for all per-vault tables, JSONL workspace trace format, retention policies, cross-platform path resolution, and the schema migration plan. It inherits the 105-tool surface from G2.1 and the architectural commitments from G2.2.

Locked decisions are summarized below. The migration file at `packages/server/src/migrations/20260519_001_initial.sql` is the authoritative source of truth for schema. `packages/server/src/schema.sql` documents the same schema for human reference and is regenerated from migrations.

## Locked decisions summary

| ID    | Decision                       | Lock                                                                                                  |
| ----- | ------------------------------ | ----------------------------------------------------------------------------------------------------- |
| D7    | Embeddings table shape         | Split: `chunks` (metadata) + `chunk_embeddings` (versioned with `is_active`) + `vec_chunks` virtual table |
| D6    | Chunk strategy                 | Hybrid heading-primary token-capped, per-vault. Defaults `heading_level=2`, `token_cap=800`, `fenced_code_atomic=true` |
| D1    | `elicit_tokens` schema         | Soft-delete via `consumed_at`, 5min TTL, 60s sweep, audit row with `proposed_change_json`             |
| D2/D5 | Idempotency TTL + tunability   | Per-vault config, `ttl_seconds=86400`, `max_result_bytes=1048576`, composite `(vault_id, key)` PK     |
| D9    | Retention policies             | Per-vault `event_log_days=30`, `traces_days=90`. Shared scheduler 60s/24h                             |
| D8    | JSONL trace format             | Full args + field redaction (`content`, `body`, `markdown`, `text`, `note_content`, `observations`), 8KB length cap |
| D4    | Memory entity materialization  | SQLite source, 1-way derived `.md`. Per-vault `memory:` block. Frontmatter banner + Relations section |
| D3    | Cross-platform paths           | Hybrid: `~/.obsidian-tc/` config + platform-native cache. `OBSIDIAN_TC_*` env overrides               |
| D10   | Migration plan + V2 audit      | Append-only `schema_migrations`, checksum-guarded, additive-only V1.x. `memory_entities` gains V2 columns |

The initial migration also folds the G2.4 B.2 schema delta on `idempotency_keys` (in-flight detection columns), since the migration has not yet been applied. See **Idempotency table** below.

## SQLite schema

One DB file per vault, located at `<cache_dir>/<vault_id>/db.sqlite`. PRAGMAs set per-connection (not in migration files): `foreign_keys = ON`, `journal_mode = WAL`.

### Migrations tracking

```sql
CREATE TABLE schema_migrations (
  version TEXT PRIMARY KEY,
  applied_at INTEGER NOT NULL,
  obsidian_tc_version TEXT NOT NULL,
  duration_ms INTEGER NOT NULL,
  checksum TEXT NOT NULL                   -- SHA-256 of migration file content
);
```

### Chunks and embeddings

```sql
CREATE TABLE chunks (
  id TEXT PRIMARY KEY,                     -- stable chunk ID, e.g. "chk_01h8x..."
  vault_id TEXT NOT NULL,
  path TEXT NOT NULL,                      -- vault-relative note path
  chunk_index TEXT NOT NULL,               -- "3" or "3.1" for sub-chunked sections
  headings TEXT NOT NULL,                  -- JSON array of heading breadcrumb
  content TEXT NOT NULL,
  content_hash TEXT NOT NULL,              -- SHA-256 of content for change detection
  token_count INTEGER NOT NULL,
  created_at INTEGER NOT NULL,
  updated_at INTEGER NOT NULL,
  -- V2-reserved (NULL in V1, populated in V2)
  cluster_id INTEGER,
  decay_score REAL,
  activation_count INTEGER DEFAULT 0,
  last_accessed_at INTEGER
);

CREATE INDEX idx_chunks_vault_path ON chunks(vault_id, path);
CREATE INDEX idx_chunks_hash ON chunks(content_hash);
CREATE INDEX idx_chunks_cluster ON chunks(cluster_id) WHERE cluster_id IS NOT NULL;

CREATE TABLE chunk_embeddings (
  chunk_id TEXT NOT NULL,
  model TEXT NOT NULL,                     -- provider:model, e.g. "openai:text-embedding-3-small"
  dimensions INTEGER NOT NULL,
  embedding BLOB NOT NULL,
  is_active INTEGER NOT NULL DEFAULT 1,    -- exactly one is_active=1 per chunk_id (app-enforced)
  generated_at INTEGER NOT NULL,
  PRIMARY KEY (chunk_id, model),
  FOREIGN KEY (chunk_id) REFERENCES chunks(id) ON DELETE CASCADE
);

CREATE INDEX idx_chunk_embeddings_active
  ON chunk_embeddings(chunk_id) WHERE is_active = 1;
CREATE INDEX idx_chunk_embeddings_model ON chunk_embeddings(model);
```

The `vec_chunks` sqlite-vec virtual table mirrors active embeddings only. Dimension count is runtime-bound at first embedding generation per vault. See **Migration runner** below for the runtime-bound creation flow.

### Workspace sessions and capture queue

```sql
CREATE TABLE workspace_sessions (
  id TEXT PRIMARY KEY,                     -- session_id, e.g. "sess_01h8x..."
  vault_id TEXT NOT NULL,
  caller TEXT,                              -- jwt sub or "anonymous"
  started_at INTEGER NOT NULL,
  ended_at INTEGER,
  trace_path TEXT NOT NULL,                 -- relative to cache_dir/<vault>/traces/
  metadata_json TEXT                        -- optional session-start metadata
);

CREATE INDEX idx_workspace_sessions_vault ON workspace_sessions(vault_id, started_at DESC);
CREATE INDEX idx_workspace_sessions_unended ON workspace_sessions(ended_at)
  WHERE ended_at IS NULL;

CREATE TABLE capture_queue (
  id TEXT PRIMARY KEY,
  vault_id TEXT NOT NULL,
  title TEXT,
  content TEXT NOT NULL,
  tags TEXT,                                -- comma-separated
  source TEXT,                              -- which client/agent captured this
  target_path_hint TEXT,                    -- destination on commit (suggestion, not binding)
  captured_at INTEGER NOT NULL,
  committed_at INTEGER,                     -- NULL until committed to vault
  committed_path TEXT                       -- final destination after commit
);

CREATE INDEX idx_capture_uncommitted ON capture_queue(vault_id, captured_at DESC)
  WHERE committed_at IS NULL;
```

### Memory entities and relations

```sql
CREATE TABLE memory_entities (
  id TEXT PRIMARY KEY,                     -- "ent_01h8x..."
  vault_id TEXT NOT NULL,
  entity_type TEXT NOT NULL,                -- person | project | decision | concept | place | other
  name TEXT NOT NULL,
  observations TEXT NOT NULL,               -- newline-separated facts
  materialize INTEGER NOT NULL DEFAULT 1,
  vault_path TEXT,                          -- materialized .md file path (relative)
  created_at INTEGER NOT NULL,
  updated_at INTEGER NOT NULL,
  -- V2-reserved (activation tracking, matches chunks)
  decay_score REAL,
  activation_count INTEGER DEFAULT 0,
  last_accessed_at INTEGER
);

CREATE INDEX idx_memory_entities_vault_type ON memory_entities(vault_id, entity_type);
CREATE INDEX idx_memory_entities_name ON memory_entities(vault_id, name);

CREATE TABLE memory_relations (
  source_id TEXT NOT NULL,
  target_id TEXT NOT NULL,
  relation_type TEXT NOT NULL,              -- arbitrary verb, e.g. "founded", "works_on"
  created_at INTEGER NOT NULL,
  PRIMARY KEY (source_id, target_id, relation_type),
  FOREIGN KEY (source_id) REFERENCES memory_entities(id) ON DELETE CASCADE,
  FOREIGN KEY (target_id) REFERENCES memory_entities(id) ON DELETE CASCADE
);

CREATE INDEX idx_memory_relations_target ON memory_relations(target_id);
```

### Idempotency table (with G2.4 in-flight detection folded in)

The G2.4 B.2 schema delta is folded into this initial CREATE TABLE since the initial migration has not yet been applied. The columns supporting in-flight detection are:

- `started_at INTEGER NOT NULL`: row insert timestamp; also serves as the canonical "this idempotency key began processing" marker. Replaces `created_at` from the original G2.3 D2 sketch since they're semantically the same in this single-execution-per-key model.
- `completed_at INTEGER`: NULL while in-flight, set on successful completion.
- `result BLOB` (nullable): NULL while in-flight, populated on completion.
- `result_size INTEGER` (nullable): NULL while in-flight, set on completion. Used by the G2.4 B.1 response-byte governor for replay sizing.

A row with `started_at IS NOT NULL AND completed_at IS NULL` is in-flight. The 60s fast-sweep reaps rows where `started_at + 60000 < now AND completed_at IS NULL` to free keys stuck after process crashes.

```sql
CREATE TABLE idempotency_keys (
  vault_id TEXT NOT NULL,
  key TEXT NOT NULL,
  tool_name TEXT NOT NULL,
  args_hash TEXT NOT NULL,                  -- 16B hex, same derivation as elicit
  started_at INTEGER NOT NULL,              -- G2.4 B.2: in-flight detection start
  completed_at INTEGER,                     -- NULL while in-flight; G2.4 B.2
  result BLOB,                              -- NULL while in-flight
  result_size INTEGER,                      -- NULL while in-flight; bytes when completed
  expires_at INTEGER NOT NULL,              -- TTL expiry (default now + 86400000ms)
  PRIMARY KEY (vault_id, key)
);

CREATE INDEX idx_idempotency_expires ON idempotency_keys(expires_at);
CREATE INDEX idx_idempotency_in_flight ON idempotency_keys(started_at)
  WHERE completed_at IS NULL;
```

### Vault edges (graph substrate)

Zero-LLM wikilink/markdown/typed-link edge store. Populated by `scan_vault_edges` (Domain 30). Used by `get_vault_edges`, `detect_vault_schema`, and the `adjacency_boost` signal in `search_vault` / `search_semantic` (Domain 6).

```sql
CREATE TABLE vault_edges (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  vault_id TEXT NOT NULL,
  source_path TEXT NOT NULL,
  target_path TEXT NOT NULL,
  edge_type TEXT NOT NULL,                  -- "wikilink" | "markdown" | "typed"
  relation_type TEXT,                       -- typed-link relation label (nullable)
  line INTEGER,                             -- source line number (nullable)
  scanned_at INTEGER NOT NULL
);

CREATE INDEX idx_vault_edges_source ON vault_edges(vault_id, source_path);
CREATE INDEX idx_vault_edges_target ON vault_edges(vault_id, target_path);
CREATE UNIQUE INDEX idx_vault_edges_unique ON vault_edges(
  vault_id, source_path, target_path, edge_type, COALESCE(relation_type, '')
);
```

## Migration runner

### File layout

```
packages/server/src/migrations/
├── 20260519_001_initial.sql
├── 20260519_001_initial.sql.checksum    # sidecar, generated at build
└── (future migrations)
```

Migration filename format: `YYYYMMDD_NNN_<slug>.sql` (or `.ts` for data-transforming migrations). Sort order is filename lexicographic, which gives chronological order naturally.

### Startup contract

For each configured vault, in vault config order:

1. Resolve DB path (`<cache_dir>/<vault_id>/db.sqlite`). Create directory if missing.
2. Open DB. Set PRAGMAs: `foreign_keys = ON`, `journal_mode = WAL`.
3. Check `schema_migrations` table existence. If missing, this is a fresh DB.
4. Discover migration files in `packages/server/src/migrations/`.
5. Read applied versions from `schema_migrations` (empty set if fresh).
6. Compute pending = available minus applied. Sort by filename.
7. For each pending migration: begin transaction, read file content, compute SHA-256, execute migration SQL (or `.ts` migration function), insert `schema_migrations` row, commit transaction.
8. For each already-applied migration: verify recorded checksum matches current file. On mismatch, refuse to start, naming the offending version, expected checksum, and actual checksum.

Failure during a migration: rollback transaction, log to stdout (event_log not safe pre-migration), exit non-zero. Surface includes the vault, version, file path, and SQLite error.

### Initial migration

`20260519_001_initial.sql` is the V1.0 baseline. Contents:

1. `CREATE TABLE schema_migrations` (the table records its own creation in the same migration).
2. All tables from the schema above, in dependency order (chunks → chunk_embeddings, memory_entities → memory_relations, vault_edges).
3. All indexes.
4. No triggers (cascades are FK-driven via `ON DELETE CASCADE`).
5. Final statement records the migration: `INSERT INTO schema_migrations VALUES ('20260519_001_initial', strftime('%s','now')*1000, '0.1.0', 0, 'sha256-placeholder');` (runner overrides `duration_ms` and `checksum` after execution).

`vec_chunks` virtual table is NOT created in this migration because the dimension count is runtime-bound (depends on per-vault `embeddings.provider` config). The first embedding generation per vault triggers `CREATE VIRTUAL TABLE vec_chunks USING vec0(chunk_id TEXT PRIMARY KEY, embedding float[$DIMS])` with the bound dimension. This is recorded as `20260519_002_vec_chunks_$DIMS` in `schema_migrations` for that vault, so each vault DB tracks its own vec0 version.

### V1.x evolution rule

Migrations within V1.x are **additive only**:

- `CREATE TABLE`
- `ALTER TABLE ADD COLUMN ... DEFAULT ...`
- `CREATE INDEX`
- `CREATE TRIGGER`
- `CREATE VIEW`

Forbidden in V1.x:

- `DROP` of any kind
- `RENAME` of any kind
- Type changes that aren't compatibility-preserving (e.g. INTEGER to TEXT)

V2.0 may be destructive. V1.x → V2.0 migration sequence is a separate ticket when V2 starts.

## JSONL trace format

### File layout

```
<cache_dir>/<vault_id>/traces/<YYYY-MM-DD>/<session_id>.jsonl
```

- `<YYYY-MM-DD>` is the session start date in UTC. Mid-session midnight does NOT rotate.
- One file per workspace session.
- Append-only. Per-session file handle held open for session lifetime.
- Flush triggers (any): 5s since last flush, 100 events buffered, `end_workspace_session`, SIGTERM.
- Crash-loss budget: ~5 seconds of events.
- Empty session files deleted on `end_workspace_session`.

### Event schema

Required fields:

```jsonc
{
  "ts": 1747595432000,           // ms epoch
  "session": "sess_01h8x...",
  "vault": "primary",
  "tool": "search_vault",
  "caller": "claude-desktop",    // jwt sub or "anonymous"
  "status": "ok",                 // "ok" | "error"
  "args_hash": "a1b2c3..."        // 16B hex
}
```

Optional fields: `args` (possibly with redacted markers), `duration_ms`, `result_summary`, `error`, `elicit_token_used`, `idempotency_key_used`, `idempotency_cache_hit`.

### Argument redaction

Default redact list (per-vault tunable): `content`, `body`, `markdown`, `text`, `note_content`, `observations`.

Length cap: `args_max_field_bytes: 8192`. Any field whose serialized length exceeds this is redacted regardless of name.

Redacted field replacement:

```jsonc
"content": {
  "redacted": true,
  "content_hash": "d4e5f6...",   // SHA-256, full 64 chars
  "content_length": 1247          // bytes of original
}
```

### Result summary shapes (per-tool examples)

```jsonc
// search_vault
"result_summary": { "chunk_count": 12, "chunk_ids": ["chk_001", "chk_044"], "modes_tried": ["text", "semantic"] }

// read_note
"result_summary": { "path": "02-projects/foo.md", "bytes": 4096, "chunk_count": 7 }

// write_note
"result_summary": { "path": "00-inbox/foo.md", "bytes_written": 1247, "chunk_ids_invalidated": ["chk_512"] }

// bulk_create_notes
"result_summary": { "created": 198, "skipped": 2, "errors": 0, "duration_ms_per_item_p50": 14 }
```

### Error shape

```jsonc
"error": {
  "code": "elicit_required",
  "message": "HITL confirmation needed for delete_note",   // 200-char max
  "details": { "elicit_token": "abc123...", "proposed_change_summary": "..." }
}
```

No stack traces in JSONL. Server-side stack traces stay in stdout / MORGIANA.

### Session lifecycle

- `start_workspace_session` opens the trace file handle, registers the row in `workspace_sessions` with `trace_path`.
- Each tool call appends one JSONL line via the held handle.
- `end_workspace_session` flushes buffer, closes handle, sets `workspace_sessions.ended_at`, deletes file if empty.
- Idle timeout (per-vault config, default 24h) auto-ends sessions.
- Server crash: unflushed buffer lost. Startup recovery pass marks orphaned sessions: `UPDATE workspace_sessions SET ended_at = ? WHERE ended_at IS NULL AND started_at < ? - 86400000`

## Retention sweeps

### TTL sweeps (60s cadence)

Single `setInterval` runs in the Bun server process. Per pass:

```sql
DELETE FROM elicit_tokens WHERE expires_at < ?;
DELETE FROM idempotency_keys WHERE expires_at < ?;
DELETE FROM idempotency_keys
  WHERE started_at + 60000 < ? AND completed_at IS NULL;   -- G2.4 B.2: reap stuck in-flight rows
```

Where `?` = `Date.now()`. All three have indexes; deletes are cheap even at thousands of rows.

### Retention sweeps (24h cadence)

Second `setInterval`. Per vault, sequential:

```sql
-- event_log
DELETE FROM event_log WHERE ts < ? - (event_log_days * 86400000);

-- workspace_sessions + JSONL files cascade
SELECT id, trace_path FROM workspace_sessions
  WHERE ended_at IS NOT NULL AND ended_at < ? - (traces_days * 86400000);
-- For each: rm -rf the trace file, then DELETE the row.

-- Walk trace folder, delete date-folders older than cutoff
-- (catches orphaned files where workspace_sessions row was already gone)
```

Final: one summary row to event_log: `{ event_type: "retention_sweep", vault, rows_deleted, traces_deleted, errors }`.

### Not auto-swept

- `chunks`, `chunk_embeddings`: source from filesystem, rebuilt on vault change.
- `chunk_embeddings` inactive rows: A/B comparison material. Manual cleanup via `reset_vault_cache --include embeddings.inactive` (V1.x slot; tool currently has `embeddings: boolean` flag in G2.1).
- `memory_entities`, `memory_relations`: SQLite source of truth, no auto-sweep. Orphan reconciliation deferred to V1.x.
- `capture_queue`: `committed_at` soft-delete only. Manual purge of committed captures via future tool (V1.x slot; not in G2.1).
- `vault_edges`: rebuilt by `scan_vault_edges` on demand. No auto-sweep; stale edges are overwritten on rescan, not accumulated.

## Cross-platform paths

### Per-OS defaults

| Component  | Linux                                                              | macOS                              | Windows                          |
| ---------- | ------------------------------------------------------------------ | ---------------------------------- | -------------------------------- |
| Config     | `~/.obsidian-tc/config.yaml`                                       | `~/.obsidian-tc/config.yaml`       | `~\.obsidian-tc\config.yaml`     |
| Cache base | `$XDG_CACHE_HOME/obsidian-tc/` or `~/.cache/obsidian-tc/`          | `~/Library/Caches/obsidian-tc/`    | `%LOCALAPPDATA%\obsidian-tc\`    |

### Override env vars

Precedence (high to low):

1. `OBSIDIAN_TC_CONFIG_DIR` (explicit config dir)
2. `OBSIDIAN_TC_CACHE_DIR` (explicit cache dir)
3. `OBSIDIAN_TC_HOME` (overrides home dir base)
4. `$HOME` / `$USERPROFILE` (OS-native)

### Startup checks

1. **No home dir + no overrides:** refuse to start. Error names `OBSIDIAN_TC_HOME` as the escape hatch.
2. **Config dir read-only:** refuse to start. First-run can create missing config dir.
3. **Cache dir read-only:** refuse to start. First-run can create missing cache dir.
4. **Symlinks:** resolved via `realpath` at startup. Canonical paths used for the session. Vault content paths (from vault config) NOT resolved.
5. **Windows long paths:** if `HKLM\SYSTEM\CurrentControlSet\Control\FileSystem\LongPathsEnabled` is 0 or missing AND worst-case path exceeds 240 chars, warn at startup. Do not refuse.

### Vault content path rules (restated from G2.2)

- Tool args carrying paths: rejected if absolute, contain `..`, contain Windows drive letters (regardless of host OS).
- Case-fold on macOS and Windows. Case-sensitive on Linux. Trailing slash normalized.
- Path normalization applied before `args_hash` derivation for idempotency and elicit tokens.

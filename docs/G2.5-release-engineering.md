## 2. Distribution channels

### 2.1 npm

Twelve package names reserved on npm. Distribution publishes eleven of them with real content (one unscoped main + one shared + one native umbrella + eight platform sub-packages). The twelfth, `@the-40-thieves/obsidian-tc`, is a defensive scope-main reservation that stays as a deprecated `0.0.1` stub for the v1.x lifespan. Lockstep version across all real-content packages.

| Package | Contents | v1.0 distribution role |
|---|---|---|
| `obsidian-tc` | Main server bin + library entry | Primary install: `npm i -g obsidian-tc` |
| `@the-40-thieves/obsidian-tc` | (empty stub) | Defensive scope-main reservation; permanent `0.0.1` stub during v1.x |
| `@the-40-thieves/obsidian-tc-shared` | TS types + Zod schemas | Plugin developers, integrators |
| `@the-40-thieves/obsidian-tc-native` | napi-rs umbrella with `optionalDependencies` selecting platform sub-package | Bundlers and direct consumers |
| `@the-40-thieves/obsidian-tc-native-linux-x64-gnu` | Prebuilt `.node` for linux-x64 (glibc) | Auto-selected via `optionalDependencies` |
| `@the-40-thieves/obsidian-tc-native-linux-arm64-gnu` | Prebuilt `.node` for linux-arm64 (glibc) | Auto-selected |
| `@the-40-thieves/obsidian-tc-native-linux-x64-musl` | Prebuilt `.node` for linux-x64 (musl/Alpine) | Auto-selected |
| `@the-40-thieves/obsidian-tc-native-linux-arm64-musl` | Prebuilt `.node` for linux-arm64 (musl/Alpine) | Auto-selected |
| `@the-40-thieves/obsidian-tc-native-darwin-x64` | Prebuilt `.node` for macOS Intel | Auto-selected |
| `@the-40-thieves/obsidian-tc-native-darwin-arm64` | Prebuilt `.node` for macOS Apple Silicon | Auto-selected |
| `@the-40-thieves/obsidian-tc-native-win32-x64-msvc` | Prebuilt `.node` for Windows x64 | Auto-selected |
| `@the-40-thieves/obsidian-tc-native-win32-arm64-msvc` | Prebuilt `.node` for Windows ARM64 | Auto-selected |

Plugin (`@the-40-thieves/obsidian-tc-plugin`) is NOT published to npm. It ships as a release artifact for sideload and through the Obsidian Community Store.

`obsidian-tc` declares `@the-40-thieves/obsidian-tc-native` as a regular dependency. `@the-40-thieves/obsidian-tc-native` declares all eight platform packages in `optionalDependencies` with `os`, `cpu`, and (for the musl variants) `libc` fields. npm install selects the right one. If selection fails (e.g., an unsupported platform with no matching triple), `@the-40-thieves/obsidian-tc-native` index falls back to pure-JS impls.

Why the scope is `@the-40-thieves` and not `@obsidian-tc`: the npm org `@obsidian-tc` does not exist; `the-40-thieves` is the verified org under the project owner's account.

### 2.2 GitHub Releases

Each tag `v{x.y.z}` produces:

1. Source tarball (auto-generated)
2. Standalone Bun binaries (4 platforms) named `obsidian-tc-{platform}-{arch}{.exe}`. Compiled via `bun build --compile --target=bun-{platform}-{arch}`.
3. Plugin .zip (`obsidian-tc-plugin-{version}.zip`) containing `main.js`, `manifest.json`, `styles.css`.
4. Docker image manifest pointer (`ghcr.io/the-40-thieves/obsidian-tc:v{x.y.z}` and `:latest`).
5. SHA256 checksums file (`SHASUMS256.txt`).

npm publishes use `--provenance` flag for attestation. Cosign binary signing and SBOM generation are deferred to v1.1.

Release notes generated from Conventional Commits log between previous tag and current.

### 2.3 Obsidian Community Plugin Store

Plugin (`obsidian-tc` companion) submitted to `obsidianmd/obsidian-releases` once per minor version. Patch updates within a minor go through the store without re-review.

Plugin version is allowed to lag server by one minor during review periods. The server announces minimum required plugin version via `/obsidian-tc/health` and refuses to operate against incompatible plugin versions with a clear error pointing to the upgrade path.

Until the first community store acceptance, BRAT (`TfTHacker/obsidian42-brat`) is the recommended sideload path. README documents both.

### 2.4 ghcr.io Docker

Multi-arch image built for `linux/amd64` and `linux/arm64`. Base image: `oven/bun:1-alpine`. No `:latest` tag floated to a non-LTS release. Image labels: OCI standard plus `org.opencontainers.image.source` pointing at the repo.

```dockerfile
FROM oven/bun:1 AS builder
WORKDIR /app
COPY package.json bun.lockb ./
COPY packages/server/package.json packages/server/
COPY packages/shared/package.json packages/shared/
COPY packages/native/package.json packages/native/

FROM oven/bun:1-alpine
COPY --from=builder /app/packages/server/dist /app
COPY --from=builder /app/node_modules /app/node_modules
WORKDIR /app
ENTRYPOINT ["bun", "run", "index.js"]
CMD ["serve"]
```

### 2.5 PyPI (V2 only)

`obsidian-tc-ml` package on PyPI. Built with `maturin` if Rust extensions creep in, otherwise pure `setuptools` + `pyproject.toml`. Pins Python `>=3.11`. Off by default. Users opt-in via `OBSIDIAN_TC_ENABLE_ML_SIDECAR=true` and a separate `pip install obsidian-tc-ml`.

## 3. Versioning policy

Lockstep across all real-content packages for v1.x. Single source of truth: root `package.json` version. Propagated to all workspace packages via `bunx bumpp` (or `npm version` if Bun's tooling lags) before tag push.

The defensive stub `@the-40-thieves/obsidian-tc` is exempt from lockstep. It stays at `0.0.1` permanently during v1.x.

Tag format: `v{major}.{minor}.{patch}`. Pre-releases use `v{major}.{minor}.{patch}-{alpha|beta|rc}.{n}`.

Plugin `manifest.json` version field matches package version at release time. The community-store-lag exception applies only between server publish and community-store acceptance; once accepted, plugin and server are tagged together.

Compatibility commitment within v1.x:
- Tool surface (the 103 tools from G2.1 r2) is additive only. No tool removal, no breaking schema change, no breaking response shape.
- Database schema migrations are additive only. No column drops, no destructive changes. New columns added with defaults. New tables added without dropping existing tables.
- Plugin REST endpoints (the 9 endpoints from G2.2) are additive only. New endpoints may be added; existing endpoint contracts remain stable.
- Config schema is additive only. New options ship with sensible defaults. No required option additions.

Breaking changes accumulate against v2.0, which is the V2 substrate unification anchor.

## 4. Native module fallback contract

Every Rust function in `packages/native/src/` ships with a TypeScript fallback in `packages/native/src/fallback.ts`. The umbrella entry point loads the platform-specific `.node` if available, otherwise the fallback module.

```typescript
// packages/native/src/index.ts
let impl: NativeBinding;
let isFallback = false;

try {
  impl = require('@the-40-thieves/obsidian-tc-native-' + getPlatformTriple());
} catch (err) {
  console.warn('[obsidian-tc native] platform binding not loaded, using pure-JS fallback. Performance will be degraded for vector and BM25 operations. Original error:', err.message);
  impl = require('./fallback');
  isFallback = true;
}

export const cosineSimilarity = impl.cosineSimilarity;
export const dotProduct = impl.dotProduct;
export const reciprocalRankFusion = impl.reciprocalRankFusion;
export const bm25Score = impl.bm25Score;
export const tokenize = impl.tokenize;
export const kmeansAssign = impl.kmeansAssign;        // V2-reserved
export const actrDecayScore = impl.actrDecayScore;     // V2-reserved
export const isNativeAvailable = () => !isFallback;
```

Performance budget for the fallback impl, enforced by benchmark suite in CI:
- `cosineSimilarity(1536-dim)`: native ≤200ns, fallback ≤1000ns (5x ceiling)
- `dotProduct(1536-dim)`: same shape as cosine
- `reciprocalRankFusion(2 lists of 100)`: native ≤50µs, fallback ≤250µs (5x ceiling)
- `bm25Score(query 8 tokens, corpus 100k chunks)`: native ≤5ms, fallback ≤50ms (10x ceiling)
- `tokenize(1000-char text, bge-small)`: native ≤200µs, fallback ≤2ms (10x ceiling)

The `[fallback]` warning is logged once per process at module load, not on every call. The `isNativeAvailable()` helper is exposed so the doctor command can report fallback state to users.

V2-reserved functions (`kmeansAssign`, `actrDecayScore`) ship as no-op throws in v1.0 with a clear error message pointing to V2 docs. Both native and fallback impls match this behavior.

## 5. CLI surface

Single `obsidian-tc` binary, Bun-compiled, with subcommands. Help surface follows `wrangler` / `supabase` / `cloudflared` patterns.

```
obsidian-tc serve [--config <path>] [--vault <id>]
obsidian-tc init [--config <path>] [--vault-path <path>]
obsidian-tc auth init [--key-path <path>]
obsidian-tc auth issue [--vault <id>] [--scopes <list>] [--ttl <duration>]
obsidian-tc auth revoke <jti>
obsidian-tc auth list
obsidian-tc auth rotate
obsidian-tc config validate [--config <path>]
obsidian-tc config show [--config <path>]
obsidian-tc migrate [--vault <id>] [--dry-run]
obsidian-tc doctor [--config <path>]
obsidian-tc version
```

Implementation notes:
- `obsidian-tc serve`: launches transport per config (STDIO if invoked as MCP client subprocess, HTTP otherwise).
- `obsidian-tc init`: writes minimal config to `~/.config/obsidian-tc/config.yaml` (path platform-resolved per G2.3). Prompts for vault path interactively unless `--vault-path` supplied. Refuses to overwrite existing config without `--force`.
- `obsidian-tc auth init`: required for `auth.mode: jwt | oauth`. Generates Ed25519 signing key at `~/.config/obsidian-tc/keys/signing.key` with mode 0600. Idempotent: refuses to overwrite existing key without `--rotate`.
- `obsidian-tc auth issue`: mints a JWT for an agent. Interactive prompts for scopes if not supplied. Writes the issued token to stdout once, with a warning that the value is not stored in plaintext anywhere.
- `obsidian-tc auth revoke`: writes a revocation entry to the auth state file. Server picks up revocations on next request via cache invalidation.
- `obsidian-tc auth list`: lists active tokens by `jti` and metadata (issued-at, scopes, vault, agent label). Does not expose token values.
- `obsidian-tc auth rotate`: rotates the signing key. Old key kept for the verification grace period (configurable in `auth.rotation.grace_seconds`, default 86400). New tokens minted with new key.
- `obsidian-tc config validate`: schema-checks config file against Zod schema. Exits non-zero on error.
- `obsidian-tc config show`: prints effective config with secrets redacted.
- `obsidian-tc migrate`: runs pending SQL migrations against each configured vault's cache.
- `obsidian-tc doctor`: runs all G2.3 self-checks (config valid, vaults reachable via REST API, plugin endpoints responding, native module status, sidecar status if enabled, disk space, cache freshness). Reports per-check status with green/yellow/red.
- `obsidian-tc version`: prints version + commit SHA + platform triple + native-or-fallback status.

### 6.3 Refusal contracts

- `obsidian-tc serve` with `auth.mode: jwt | oauth` and no signing key → exit 1, message points at `auth init`.
- `obsidian-tc serve` with `auth.mode: none` bound to non-localhost → exit 1.
- `obsidian-tc auth issue` with no signing key → exit 1, message points at `auth init`.
- `obsidian-tc auth init` with key already present → exit 1 unless `--rotate`.

## 8. MCP resource primitive deferral

v1.0 ships tools-only. MCP resources, prompts, and sampling primitives are not implemented in v1.0.

Rationale: the MCP spec for resources is shifting (resource template URIs, subscription semantics, mime types). Shipping a partial impl risks breaking change at minor-version boundary.

Target slot: v1.1 or v1.2. Pre-condition: MCP spec for resources reaches stability in the upstream spec repo.

`ROADMAP.md` documents the deferral and the trigger condition.

## 9. Release runbook

Steps from version bump to public availability:

1. **Branch** off `main` to `release/v{x.y.z}` (only required for major or minor releases; patch releases can ship directly from main if hot-fix scope).
2. **Bump version**: `bunx bumpp` (interactive, asks major/minor/patch, propagates to all workspace `package.json` files and plugin `manifest.json`). The defensive stub `@the-40-thieves/obsidian-tc` is excluded from the bump.
3. **Update CHANGELOG.md**: `bunx changelogen` (auto-generates from Conventional Commits since last tag, edit for highlight section).
4. **Open PR** from `release/v{x.y.z}` to `main`. CI must pass on all four workflows (`ci-server`, `ci-plugin`, `ci-native` — a 3-cell host matrix plus a build-only musl cross-compile job, `ci-integration`).
5. **Merge PR** with merge commit (not squash, to preserve commit history for changelog regeneration).
6. **Tag** at the merge commit: `git tag v{x.y.z} && git push --tags`.
7. **`publish.yml` fires** automatically on tag push:
   - Builds native module on each of 8 platforms in parallel (linux-arm64 + win32-arm64 cross-compiled on x64 runners; linux musl via `napi build -x`).
   - Publishes 8 `@the-40-thieves/obsidian-tc-native-{platform}` sub-packages to npm.
   - Publishes `@the-40-thieves/obsidian-tc-native` umbrella to npm.
   - Publishes `@the-40-thieves/obsidian-tc-shared` to npm.
   - Publishes `obsidian-tc` (main, unscoped) to npm.
   - Does NOT publish to `@the-40-thieves/obsidian-tc`; it stays at the `0.0.1` deprecated stub.
   - All npm publishes use `--provenance` flag for npm attestation.
   - Builds standalone Bun binaries for 4 platforms.
   - Builds plugin zip.
   - Generates SHA256 checksums.
   - Builds and pushes Docker image to `ghcr.io/the-40-thieves/obsidian-tc:v{x.y.z}` and `:latest` (latest moved only for non-pre-release tags).
   - Drafts GitHub Release with all artifacts attached and changelog body.
8. **Manual review** of draft GitHub Release. Publish when satisfied.
9. **Plugin store submission**: open PR to `obsidianmd/obsidian-releases` against `community-plugins.json` for new minor versions. Patch versions auto-flow through.
10. **Announce**: post to `#obsidian-tc-releases` (vault-internal Linear channel), pinned tweet draft if major.

Rollback path for npm: deprecate (`npm deprecate obsidian-tc@{x.y.z} "rollback"`). Do not unpublish. Tag `v{x.y.z+1}` with the fix.

Rollback path for Docker: previous `:v{x.y.z-1}` tag remains; users pin to it. `:latest` re-pointed to last known good.

Rollback path for plugin store: revert PR to `obsidianmd/obsidian-releases` if regression caught within review window. After release, push patch.

## 10. CI/CD workflows

Four workflows under `.github/workflows/`. Each is gated by path filters where appropriate to minimize wasted CI cost. See the actual workflow files in `.github/workflows/` for the executable definitions; the sections below describe the design intent.

### 10.1 `ci-server.yml`

Matrix on `{ubuntu, macos, windows}` x `{bun 1.1.x, latest}`. Steps: install (frozen lockfile), lint/typecheck/test/build for `shared` and `server` packages, then bench on the latest-Bun + Ubuntu cell. Separate `fallback-test` job runs the full server test suite with `OBSIDIAN_TC_FORCE_FALLBACK=true` to verify the pure-JS path keeps parity with native.

### 10.2 `ci-plugin.yml`

Ubuntu-only. Install, lint, typecheck, build plugin. `verify-manifest` script asserts that `manifest.json` version matches the root package version and that all required Obsidian community-store fields are present. Plugin artifact (`main.js`, `manifest.json`, `styles.css`) uploaded for downstream consumption.

### 10.3 `ci-native.yml`

Three-cell host build matrix, kept lean for PR-queue throughput (the macOS hosted pool is capped at 5 concurrent jobs account-wide): `x86_64-unknown-linux-gnu` (linux-x64-gnu), `aarch64-apple-darwin` (darwin-arm64), `x86_64-pc-windows-msvc` (win32-x64-msvc). The other release triples build only in the publish matrix. Each cell: rust toolchain install, rust-cache, build the `.node` binding, an F1 loader-clobber guard, cargo tests, upload prebuild artifact. A separate build-only job cross-compiles the two linux-musl targets via `napi build -x` (cargo-zigbuild + zig) to validate that path without publishing.

### 10.4 `publish.yml`

Tag-triggered (`tags: ['v*']`). Pipeline: build-native (8-platform matrix; linux-arm64 + win32-arm64 cross-compiled, linux musl via napi -x) -> publish-npm (assemble platform sub-packages, publish all real-content packages with `--provenance` for npm attestation) -> build-binaries (4-platform Bun standalone) -> build-plugin -> build-docker (linux/amd64 + linux/arm64) -> draft-release (SHA256 checksums, GitHub Release in draft state for manual review). Cosign binary signing and CycloneDX SBOM deferred to v1.1. The defensive stub `@the-40-thieves/obsidian-tc` is NOT published by this workflow; it stays at the `0.0.1` deprecated stub.

## 13. Carry-forwards to G3 (Simplicity Check)

Items surfaced during G2.5 drafting that warrant audit at the Simplicity Check gate.

1. **Prebuild matrix size**: do we need linux-arm64 in v1.0 if no target user owns ARM Linux yet? Possible drop.
2. **Workflow split**: could the four workflows (`ci-server`, `ci-plugin`, `ci-native`, `publish`) consolidate to two (`ci.yml` matrix + `publish.yml`) without losing parallelism?
3. **`obsidian-tc init` scope**: is the command earning its keep, or is "copy `config.example.yaml`" sufficient?
4. **Nine npm package names reserved**: 1 unscoped main + 1 scoped defensive stub + 1 shared + 1 native umbrella + 5 platform sub-packages = 9 total. v1.0 publishes 8 with real content; the scoped defensive stub stays a permanent `0.0.1` reservation. Audit candidates: (a) could `@the-40-thieves/obsidian-tc-native` host all binaries directly and skip the 5 platform sub-packages (smaller install matrix at cost of larger artifact)? (b) should `@the-40-thieves/obsidian-tc` ship a real mirror release at v1.0 instead of staying a stub?
5. **Plugin distribution dual-path**: GitHub Releases AND Community Store. Is the maintenance overhead worth it for v1.0, or is one channel enough?
6. **Docker image base**: `oven/bun:1-distroless` is current at design time but may not be stable. Alternative: `oven/bun:1-alpine` if distroless variant lags.
7. **SBOM generation**: CycloneDX via `cdxgen` is the choice. Alternative: SPDX via `syft`. G3 audits whether dual-format ships or single.
8. **Cosign keyless signing**: keyless via GitHub OIDC is the choice. Alternative: keyed via Fulcio. G3 audits trust assumptions.
